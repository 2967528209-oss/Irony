# 错误处理与重试策略

## 架构视角：为什么 LLM 调用需要特殊的错误处理

传统 API 调用失败通常是代码 Bug 或服务宕机。但 LLM API 有独特的失败模式：

1. **Rate Limit（429）**— 最常见，不是错误而是流量控制，等一下就好
2. **Timeout** — LLM 推理慢（几秒到几十秒），比普通 API 慢 10-100 倍
3. **上游不稳定** — LLM 提供商经常出现间歇性故障
4. **非确定性输出** — 同样输入可能产生不同质量的输出

所以 LLM 调用层需要设计为「**默认不可靠**」，通过重试、降级、熔断来保证可用性。

## 错误分类与应对策略

```
LLM API 错误
├── 可重试（Retryable）
│   ├── 429 Rate Limit    → 指数退避重试
│   ├── 500 Server Error  → 短暂等待后重试
│   ├── 502/503 网关错误   → 切换到备用模型
│   └── Timeout           → 增加超时时间后重试
│
├── 不可重试（Non-retryable）
│   ├── 400 Bad Request   → 修复请求参数
│   ├── 401 Unauthorized  → 检查 API Key
│   ├── 403 Forbidden     → 检查权限/额度
│   └── 404 Model Not Found → 检查模型名称
│
└── 内容层错误（Logical）
    ├── 输出被内容审核截断  → 调整 Prompt
    ├── JSON 解析失败      → 重试或换 Prompt
    └── 幻觉/不相关输出    → 重试 + 验证逻辑
```

## 重试策略

### 指数退避（Exponential Backoff）

核心思想：每次重试等待时间翻倍，避免大量请求同时涌入。

```
第 1 次重试：等 1 秒
第 2 次重试：等 2 秒
第 3 次重试：等 4 秒
第 4 次重试：等 8 秒（到达上限，停止重试）
```

加入随机抖动（Jitter）避免多个客户端同步重试：

```
实际等待 = min(baseDelay × 2^retryCount, maxDelay) + random(0, 1000)ms
```

### 重试决策矩阵

| 错误类型 | 是否重试 | 最大次数 | 退避策略 |
|----------|---------|---------|---------|
| 429 Rate Limit | 是 | 5 | 指数退避 + 使用 Retry-After 头 |
| 500 Internal Error | 是 | 3 | 指数退避 |
| 502/503 Gateway | 是 | 3 | 指数退避 + 可切换提供商 |
| Timeout | 是 | 2 | 延长超时时间 |
| 400 Bad Request | 否 | — | 直接抛出 |
| 401/403 Auth | 否 | — | 直接抛出 |

## 降级策略（Fallback）

当主模型持续失败时，自动降级到备用方案：

```
降级链路：
DeepSeek V4 Pro (首选)
  → DeepSeek V3 (性能降级但更稳定)
    → 本地模型 Ollama (完全离线)
      → 缓存响应 (最后兜底)
        → 友好的错误提示
```

### 架构模式：模型路由器

```
用户请求
    ↓
[ 模型路由器 ]
    ├── 正常 → 主模型（V4 Pro）→ 返回
    ├── 429  → 等待重试 → 主模型 → 返回
    ├── 500  → 重试 3 次 → 失败 → 降级到备用模型 → 返回
    └── 全部失败 → 返回缓存/错误提示
```

## 熔断器模式（Circuit Breaker）

当某个 LLM 提供商连续失败 N 次后，暂时「熔断」——不再发请求，直接降级。过一段时间后尝试「半开」恢复。

```
状态机：
  关闭（正常）──[连续 5 次失败]──→ 打开（熔断）
       ↑                              ↓
       └──[试探成功]── 半开 ←──[等 30 秒]
```

作用：避免在提供商故障期间浪费请求和等待时间。

## 超时管理

LLM 请求的超时设置应该比普通 API 长得多：

| 场景 | 推荐超时 | 原因 |
|------|---------|------|
| 简短回复（< 100 tokens） | 15 秒 | 含首字延迟 |
| 中等回复（100-500 tokens） | 30 秒 | 生成时间线性增长 |
| 长文生成（> 500 tokens） | 60-120 秒 | 大量 token 生成 |
| 流式请求 | 总超时 120 秒 + 单 chunk 超时 30 秒 | 流式要监控两个维度 |

## 架构决策总结

| 决策点 | 建议 | 原因 |
|--------|------|------|
| 重试次数 | 最多 3-5 次 | 太多会阻塞用户 |
| 退避基数 | 1 秒 | 太短无效，太长影响体验 |
| 是否降级 | 是 | 可用性优先于最优质量 |
| 是否熔断 | 生产环境必须 | 防止级联故障 |
| 超时时间 | 比普通 API 长 3-5 倍 | LLM 推理本身就慢 |
