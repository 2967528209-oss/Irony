## 各层职责

### 1. 前端展示层

**职责**：管理对话 UI 状态，处理流式渲染和用户交互。

| 关注点 | 解决方案 |
|--------|---------|
| 消息列表状态 | `useChat()` 自动管理 messages 数组 |
| 流式渲染 | SSE + 增量追加文本 |
| 打字机效果 | 逐 chunk 渲染 `delta.content` |
| 中断生成 | `AbortController.abort()` |
| 加载状态 | `isLoading` 标志位 |
| 错误展示 | 友好的错误提示 + 重试按钮 |

### 2. API 路由层

**职责**：接收前端请求，校验参数，调用 LLM，返回流式响应。

```
请求进入 → Zod 参数校验 → 鉴权检查 → 调用 LLM 层 → 流式返回
```

关键设计：
- API 路由本身**不关心**底层用的是哪个模型
- 通过 `streamText()` 返回标准 SSE 流
- 请求/响应格式统一，前端无感知切换

### 3. LLM 调用层（核心）

**职责**：模型选择、错误处理、成本控制。这是整个架构最核心的一层。

四大组件：

| 组件 | 职责 | 关键指标 |
|------|------|---------|
| **模型路由器** | 根据任务/成本/可用性选择模型 | 路由准确率、成本节省率 |
| **重试引擎** | 指数退避重试、自动降级 | 重试成功率、平均延迟 |
| **熔断器** | 屏蔽持续故障的 Provider | 熔断触发次数、恢复速度 |
| **Token 计量器** | 统计用量、成本核算、预算控制 | 每用户/每请求的 token 消耗 |

### 4. Provider 适配层

**职责**：屏蔽各 LLM 提供商的 API 差异，提供统一接口。

```
统一接口：generateText({ model, messages, ... })

内部适配：
├── OpenAI 兼容（DeepSeek、OpenAI、Groq）→ openai SDK
├── Anthropic → @anthropic-ai/sdk
└── 本地模型 → Ollama（OpenAI 兼容）
```

**核心价值**：切换 Provider 只需改配置，业务代码零修改。

## 数据流详解

### 非流式请求

```
前端 POST → API Route → streamText() → Provider → LLM 推理
                                                      ↓
前端 ← JSON 响应 ← API Route ← 完整文本 ← Provider ← 推理完成
```

### 流式请求

```
前端 POST ────→ API Route ────→ Provider ────→ LLM 推理开始
                                                    ↓
前端 ← chunk 1 ← API Route ← delta ← Provider ← token 1
前端 ← chunk 2 ← API Route ← delta ← Provider ← token 2
  ...                                              ...
前端 ← [DONE]  ← API Route ← stop  ← Provider ← 推理完成
```

### 错误处理流

```
前端 POST → API Route → Provider → LLM
                                    ↓ 429 Rate Limit
                         重试引擎 → 等 1s → 重试 → 成功 → 返回
                                    ↓ 再次 429
                         重试引擎 → 等 2s → 重试 → 成功 → 返回
                                    ↓ 连续失败
                         熔断器触发 → 降级到备用模型 → 返回
                                    ↓ 全部失败
                         返回错误 → 前端展示重试按钮
```

## Token 经济学

### 成本结构

```
单次调用成本 = Prompt tokens × 输入价格 + Completion tokens × 输出价格
```

### 成本优化杠杆

| 杠杆 | 效果 | 实现 |
|------|------|------|
| 模型降级 | 成本降 5-10x | 简单任务用小模型 |
| Prompt 压缩 | 减少输入 tokens | 对话历史摘要/截断 |
| 缓存 | 避免重复调用 | 相同 prompt 缓存结果 |
| 上下文截断 | 控制增长 | 固定窗口 / token 预算 |
| 批量请求 | 降低请求开销 | Embedding 批量调用 |

### 多轮对话的 Token 增长

```
第 1 轮：system(10) + user(15)                    = 25 prompt tokens
第 2 轮：system(10) + user(15) + AI(22) + user(12) = 59 prompt tokens
第 3 轮：+ AI(33) + user(16)                       = 108 prompt tokens
第 N 轮：O(N²) 增长 ← 这就是必须做上下文管理的原因
```

## 关键架构决策

| 决策 | 推荐 | 原因 |
|------|------|------|
| 是否用 AI SDK | 全栈项目用 | 省大量胶水代码 |
| Provider 几个 | 至少 2 个 | 一个主力 + 一个降级 |
| 本地模型要不要 | 开发环境推荐 | 省钱 + 离线可用 |
| 重试策略 | 指数退避 + 熔断 | 标准工业实践 |
| 上下文管理 | token 预算截断 | 比消息数截断更精确 |
| Streaming | 默认开启 | 用户体验远优于等待 |

## 技术选型总结

```
全栈 AI 应用的 LLM 调用层最小可行架构：

1. Vercel AI SDK (ai + @ai-sdk/openai)  → 统一接口
2. DeepSeek API                          → 主力 Provider
3. 指数退避重试                           → 基础容错
4. Token 用量统计                         → 成本可见
5. 对话历史截断                           → 上下文管理
```

以上 5 点覆盖了 90% 的场景。随着业务增长，再逐步加入熔断器、模型路由、缓存等高级组件。
