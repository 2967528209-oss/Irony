# RAG 架构全景

## RAG 是什么

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思想：

> **不要让 LLM 凭记忆回答问题，而是先帮它检索到相关资料，再让它基于资料生成回答。**

解决的核心问题：
- LLM 知识有截止日期（训练数据滞后）
- LLM 会产生幻觉（自信地胡说八道）
- LLM 不了解你的私有数据（内部文档、代码库）

## 最简 RAG 流程

```
用户提问 "什么是公司的请假政策？"
         ↓
  1. Embedding → 将问题转为向量
         ↓
  2. 检索 → 从向量数据库中找到最相似的文档片段
         ↓
  3. 拼接 → 将检索到的文档 + 用户问题组合成 Prompt
         ↓
  4. 生成 → LLM 基于提供的资料生成回答
         ↓
  "根据公司制度，年假为15天，需提前3天申请..."（附来源引用）
```

## RAG 演进三阶段

### Naive RAG（基础版）

```
文档 → 分块 → Embedding → 存入向量库
                                    ↓
用户提问 → Embedding → 向量检索 → Top-K 结果 → 拼入 Prompt → LLM 生成
```

**特点**：简单直接，能跑起来
**问题**：
- 检索质量不稳定（用户问法多样）
- 分块粗暴导致上下文断裂
- 无法处理复杂查询（多跳推理、对比分析）

### Advanced RAG（进阶版）

在 Naive RAG 基础上增加三个环节：

```
       ┌── Pre-retrieval（检索前优化）
       │   ├── Query 改写（让问题更适合检索）
       │   ├── Query 分解（复杂问题拆成子问题）
       │   └── HyDE（先生成假答案再检索）
       │
用户提问 ──→ 检索 ──→ Post-retrieval（检索后优化）──→ 生成
                      ├── 重排序（Reranker 精排）
                      ├── 结果过滤（去重、去噪）
                      └── 上下文压缩（提取关键段落）
```

**核心改进**：检索前优化查询 + 检索后精排结果

### Modular RAG（模块化）

将 RAG 拆成可插拔的模块，像乐高一样组装：

```
┌─────────────────────────────────────────────────┐
│                  编排层 (Orchestrator)            │
│                                                  │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌───────┐ │
│  │ 路由器  │  │ 判断器  │  │ 自适应  │  │ 融合  │ │
│  │ Router │  │ Judge  │  │ 检索器  │  │Fusion│ │
│  └────────┘  └────────┘  └────────┘  └───────┘ │
│                                                  │
├──────────────────────────────────────────────────┤
│                  模块层                           │
│                                                  │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌───────┐ │
│  │ 分块器  │  │Embedder│  │检索器   │  │重排序  │ │
│  │Chunker │  │        │  │Retriever│ │Reranker│ │
│  └────────┘  └────────┘  └────────┘  └───────┘ │
│  ┌────────┐  ┌────────┐  ┌────────┐            │
│  │ 生成器  │  │ 评估器  │  │引用追溯 │            │
│  │Generator│ │Evaluator│ │Citation│            │
│  └────────┘  └────────┘  └────────┘            │
└──────────────────────────────────────────────────┘
```

**核心理念**：每个模块独立、可替换、可组合。

## 完整 RAG 系统架构

```
┌──────────────────────────────────────────────────────────────┐
│                        离线管线 (Indexing Pipeline)            │
│                                                              │
│  数据源          文档处理           Embedding          存储    │
│ ┌──────┐     ┌──────────┐     ┌──────────┐     ┌────────┐  │
│ │ PDF  │     │ 加载/解析  │     │ 文本→向量  │     │向量数据库│  │
│ │ MD   │ ──→ │ 清洗/分块  │ ──→ │ 批量编码  │ ──→ │pgvector│  │
│ │ HTML │     │ 元数据提取  │     │          │     │        │  │
│ │ 代码 │     │           │     │          │     │        │  │
│ └──────┘     └──────────┘     └──────────┘     └────────┘  │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│                       在线管线 (Query Pipeline)               │
│                                                              │
│  用户提问                                                     │
│     ↓                                                        │
│  ┌──────────────┐                                            │
│  │ Query 理解    │  改写 / 分解 / 意图识别                      │
│  └──────┬───────┘                                            │
│         ↓                                                     │
│  ┌──────────────┐                                            │
│  │ 检索          │  向量检索 + 关键词检索（混合）                 │
│  └──────┬───────┘                                            │
│         ↓                                                     │
│  ┌──────────────┐                                            │
│  │ 重排序 & 过滤 │  Reranker 精排 + 去重 + 相关性过滤            │
│  └──────┬───────┘                                            │
│         ↓                                                     │
│  ┌──────────────┐                                            │
│  │ 生成          │  LLM 基于检索结果 + 原始问题生成回答           │
│  └──────┬───────┘                                            │
│         ↓                                                     │
│  ┌──────────────┐                                            │
│  │ 引用标注       │  标注回答中每句话的来源文档                   │
│  └──────────────┘                                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 两条管线的分离

RAG 系统本质上是**两条独立的管线**：

| 管线 | 触发时机 | 耗时 | 关注点 |
|------|---------|------|--------|
| **Indexing**（离线） | 文档导入/更新时 | 分钟~小时 | 分块质量、Embedding 覆盖率 |
| **Query**（在线） | 用户每次提问时 | 秒级 | 检索精度、响应速度 |

这意味着：
- Indexing 可以异步、批量、不着急
- Query 必须快、准、实时

## 核心指标

| 指标 | 含义 | 目标 |
|------|------|------|
| **Recall@K** | Top-K 结果中包含正确答案的比例 | > 90% |
| **Precision@K** | Top-K 结果中相关文档的比例 | > 70% |
| **Faithfulness** | 回答是否忠于检索到的文档 | > 95% |
| **Answer Relevance** | 回答是否切中用户问题 | > 85% |
| **Latency** | 从提问到首字输出的时间 | < 3 秒 |

## 各环节的技术选择（速查）

| 环节 | 技术选项 | 本学习路径首选 |
|------|---------|--------------|
| 文档解析 | LangChain Loaders / Unstructured | LangChain |
| 分块 | 固定大小 / 递归 / 语义 | 递归分块 |
| Embedding | OpenAI / BGE / Cohere | DeepSeek Embedding |
| 向量存储 | pgvector / Pinecone / Chroma | pgvector |
| 检索 | 向量 / 关键词 / 混合 | 混合检索 |
| 重排序 | Cohere Reranker / BGE Reranker | Cohere |
| Query 改写 | HyDE / Multi-query | Multi-query |
| 评估 | RAGAS / DeepEval | RAGAS |
