# Query 改写与分解

## 问题本质

用户的提问往往不适合直接检索：

```
用户问："这个系统怎么用？"
→ 太模糊，检索不到有用文档

用户问："对比 A 和 B 的优缺点"
→ 一个问题需要检索两个主题

用户问："为什么上周部署失败了？"
→ 需要先理解背景再检索
```

Query 改写在**检索之前**优化查询，让检索结果更精准。

## 主要策略

### 1. Query 改写（Rewriting）

让 LLM 把用户问题改写为更适合检索的形式：

```
用户原始：  "这个系统怎么用？"
改写后：    "系统使用指南 操作手册 快速入门"
```

### 2. Multi-query（多查询）

一个问题生成多个变体，分别检索后合并结果：

```
原始问题："pgvector 和 Pinecone 哪个好？"

生成变体：
  Q1: "pgvector 的优势和特点"
  Q2: "Pinecone 的优势和特点"
  Q3: "向量数据库对比评测"

三次检索 → 合并去重 → Top-K
```

**效果**：覆盖面更广，召回率显著提升。

### 3. HyDE（假设性文档嵌入）

先让 LLM 生成一个「假答案」，用假答案去检索：

```
用户问题：  "什么是 HNSW 索引？"
         ↓
LLM 生成假答案：  "HNSW 是一种基于图的近似最近邻搜索算法，
                  它构建多层导航图..."
         ↓
用假答案的 Embedding 去检索
         ↓
检索到的真实文档通常更相关
```

**原理**：假答案和真实文档在表述风格上更接近（都是陈述句），比用疑问句检索效果更好。

### 4. Step-back Prompting（退后一步）

对具体问题，先退后到更通用的问题：

```
具体问题：  "Next.js 14 的 App Router 怎么配置中间件？"
退后问题：  "Next.js App Router 中间件的工作原理和配置方式"
```

### 5. Query 分解（Decomposition）

复杂问题拆成子问题，分别检索后综合回答：

```
复杂问题：  "设计一个支持百万用户的 RAG 系统需要考虑什么？"

分解为：
  子问题 1: "RAG 系统的基本架构组件"
  子问题 2: "向量数据库如何支持大规模数据"
  子问题 3: "高并发场景下的 RAG 性能优化"
  子问题 4: "RAG 系统的成本控制策略"
```

## 策略选择

| 策略 | 适用场景 | 成本 | 效果提升 |
|------|---------|------|---------|
| Query 改写 | 模糊/口语化查询 | 1 次 LLM 调用 | 中 |
| Multi-query | 多角度/多主题 | 1 次 LLM + N 次检索 | 高 |
| HyDE | 概念性/定义性问题 | 1 次 LLM + 1 次检索 | 中高 |
| Step-back | 具体技术问题 | 1 次 LLM | 中 |
| Query 分解 | 复杂多跳问题 | 1 次 LLM + N 次检索 | 高 |

## 架构决策

| 决策 | 建议 | 原因 |
|------|------|------|
| 默认策略 | Multi-query | 通用性强，效果好 |
| 是否总是改写 | 先判断意图再决定 | 简单明确的问题不需要改写 |
| 与 Reranker 配合 | Multi-query + Reranker | 扩大召回 + 精确排序 |
