# RAG 系统 — 整体架构理解

## 全景架构

```
┌──────────────────────────────────────────────────────────────────┐
│                     离线管线 (Indexing Pipeline)                   │
│                                                                  │
│  数据源        文档处理        分块          Embedding      向量库  │
│ ┌──────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌──────┐ │
│ │PDF/MD│   │加载/解析  │   │递归分块   │   │文本→向量  │   │pg    │ │
│ │HTML  │──→│清洗      │──→│500-1000字│──→│批量编码  │──→│vector│ │
│ │代码  │   │元数据提取 │   │重叠50-200│   │归一化    │   │+HNSW│ │
│ └──────┘   └─────────┘   └─────────┘   └─────────┘   └──────┘ │
│                                                                  │
├──────────────────────────────────────────────────────────────────┤
│                     在线管线 (Query Pipeline)                     │
│                                                                  │
│  ┌─────────┐   ┌─────────┐   ┌──────────┐   ┌──────────────┐  │
│  │ Query    │   │ 检索     │   │ 重排序    │   │ 生成          │  │
│  │ 改写/分解 │──→│ 混合检索 │──→│ Reranker │──→│ LLM + 引用   │  │
│  │ Multi-Q  │   │ 向量+BM25│   │ Top-20→5 │   │ 结构化输出   │  │
│  └─────────┘   └─────────┘   └──────────┘   └──────────────┘  │
│                                                                  │
├──────────────────────────────────────────────────────────────────┤
│                     评估层 (Evaluation)                           │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌───────────────────────┐       │
│  │ 评估数据集 │   │ RAGAS    │   │ 指标看板               │       │
│  │ Q + A     │──→│ 自动评估 │──→│ Recall / Faithfulness │       │
│  └──────────┘   └──────────┘   └───────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 实践项目映射

我们的 rag-app 实际实现了上述架构的核心部分：

```
实践项目架构：

01-indexing.ts（Indexing 管线）
  Markdown 文件 → cleanMarkdown() 清洗 → recursiveChunk() 递归分块
    → Ollama bge-m3 Embedding → JSON 文件存储

02-search.ts（检索核心）
  查询 → bge-m3 Embedding → cosineSimilarity() 遍历比对 → Top-K

03-rag-chat.ts（完整 RAG）
  检索 Top-3 + 拼接 Prompt + DeepSeek 流式生成 + [来源N] 引用标注

04-interactive.ts（交互入口）
  命令行 REPL → 缓存向量库 → 实时问答
```

对照全景架构，实践项目覆盖了核心链路，**未实现**的部分是进阶优化：

| 环节 | 全景架构 | 实践项目 | 差距 |
|------|---------|---------|------|
| 文档加载 | 多格式（PDF/HTML/代码） | 仅 Markdown | 缺 PDF 解析器 |
| 清洗 | 按格式定制清洗规则 | 去 ASCII 图表 + 表格转文本 | 基本够用 |
| 分块 | 递归分块 | 递归分块（800 字/100 重叠） | 已实现 |
| Embedding | 云端 API / 本地模型 | Ollama bge-m3 本地 | 已实现 |
| 向量存储 | pgvector + HNSW 索引 | JSON 文件 + 遍历 | 无索引，O(n) |
| 检索 | 混合检索 + RRF 融合 | 纯向量检索 | 缺关键词检索 |
| 重排序 | Cross-encoder Reranker | 无 | 缺精排 |
| Query 优化 | Multi-query / HyDE | 无 | 缺查询改写 |
| 生成 | 流式 + 引用标注 | 流式 + [来源N] | 已实现 |
| 评估 | RAGAS 自动评估 | 人工观察 | 缺自动化评估 |

## 实践中的踩坑与架构洞察

### 1. Embedding 模型的语言适配

**踩坑**：nomic-embed-text 对中文文档的检索完全失效——所有查询返回相同结果，相似度挤在 0.69-0.71 之间无区分度。

**洞察**：Embedding 模型的语言能力是 RAG 质量的地基。选型时必须验证目标语言的效果。

```
模型选择决策树：
  纯英文内容 → nomic-embed-text (768d, 轻量)
  中文/多语言 → bge-m3 (1024d, 专为多语言设计)
  追求质量    → OpenAI text-embedding-3-large (3072d)
  追求隐私    → 本地模型 (Ollama)
```

### 2. 文档清洗决定检索上限

**踩坑**：ASCII 图表（`│┌─└` 等字符）进入 Embedding 后产生噪音向量，导致图表块总是被检索到。

**洞察**：Indexing 管线的第一定律是「垃圾进，垃圾出」。清洗策略需要根据数据特点定制：
- 技术文档：去除 ASCII 图表，保留代码示例
- Markdown：表格转纯文本，压缩空行
- 通用规则：过滤 < 80 字符的短块

### 3. 分块粒度的权衡

**踩坑**：初始 500 字分块 + 无最小长度过滤 → 产生 23 字的标题块，这种块的 Embedding 过于泛化，匹配所有查询。

**洞察**：

```
块太小（< 100 字）→ 语义信息不足，向量泛化
块太大（> 1500 字）→ 噪音多，相关性被稀释
块合理（400-800 字）→ 信息完整，语义集中

关键参数：
  chunk_size: 800（主体大小）
  overlap: 100（块间重叠防信息切断）
  min_size: 80（过滤无意义的短块）
```

### 4. 双模型架构的合理性

**实践验证**：Embedding 和生成使用不同模型是生产中的标准做法。

```
我们的架构：
  Embedding: Ollama bge-m3（本地，免费，快速）
  生成:      DeepSeek API（云端，强力，流式）

生产级架构通常也是分离的：
  Embedding: 专用小模型（成本低，速度快）
  生成:      大语言模型（能力强，成本高）
```

这样设计的原因：Embedding 需要的是「理解语义」，不需要「生成文本」；而 LLM 正好相反。用同一个模型做两件事反而不如各用专长。

## 每个环节的核心职责

| 环节 | 输入 | 输出 | 核心技术 | 质量影响 |
|------|------|------|---------|---------|
| 文档处理 | 原始文件 | 清洁文本 + 元数据 | 解析器 + 清洗规则 | 垃圾进垃圾出 |
| 分块 | 长文本 | 文档块列表 | 递归分块 | 决定检索粒度 |
| Embedding | 文本块 | 向量 | Embedding 模型 | 决定语义匹配能力 |
| 向量存储 | 向量 + 元数据 | 可检索的索引 | pgvector + HNSW | 检索速度 |
| Query 改写 | 用户原始问题 | 优化后的查询 | LLM | 扩大检索召回 |
| 检索 | 查询向量 | Top-K 候选 | 混合检索 + RRF | 召回率 |
| 重排序 | Top-K 候选 | 精排 Top-N | Cross-encoder | 精度 |
| 生成 | 问题 + 文档 | 回答 + 引用 | LLM + Prompt | 最终质量 |
| 评估 | 系统输出 | 质量指标 | RAGAS | 优化方向 |

## 从 Naive 到 Production 的升级路径

### Level 1：能跑（Naive RAG）← 我们的实践项目在这里

```
文档 → 递归分块 → Embedding → JSON 存储 → 向量检索 → LLM
```
- 核心链路完整
- 能回答基础问题
- 局限：无精排、无查询优化、无关键词检索

### Level 2：好用（Advanced RAG）

在 Level 1 基础上增加：

```
+ 混合检索（向量 + BM25 关键词互补）
+ Reranker（Cross-encoder 精排 Top-20 → Top-5）
+ Query 改写（Multi-query 扩大召回）
+ 引用标注完善（结构化输出 + 前端跳转）
+ 相似度阈值过滤（去掉 < 0.3 的无关结果）
```

实际效果提升：检索 Recall 从 ~70% 提升到 ~90%

### Level 3：生产级（Production RAG）

在 Level 2 基础上增加：

```
+ pgvector 替代 JSON（HNSW 索引，支持百万级向量）
+ 增量索引（文档变更时只更新差异部分）
+ 评估管线（RAGAS 自动回归测试）
+ 权限控制（多租户数据隔离）
+ 缓存层（热门查询结果缓存）
+ 可观测性（Langfuse 追踪每次调用）
```

## 进阶架构：超越基础 RAG

### Agentic RAG

让 Agent 决定「是否需要检索」以及「怎么检索」：

```
用户提问
  ↓
Agent 判断：
  ├── 简单问候 → 直接回答，不检索
  ├── 事实性问题 → 标准 RAG 检索
  ├── 复杂问题 → 分解为子问题，多次检索
  └── 需要计算 → 调用工具而非检索
```

与标准 RAG 的区别：标准 RAG 是「固定流程」，Agentic RAG 是「动态决策」。

### Graph RAG

用知识图谱增强检索：

```
标准 RAG：
  "谁是 CEO？" → 检索包含 CEO 的文档 → 回答

Graph RAG：
  "谁是 CEO？" → 查询知识图谱 (公司) -[CEO]→ (张三) → 回答
  "CEO 和 CTO 什么关系？" → 图谱遍历关系链 → 回答
```

适用：实体关系密集的场景（组织架构、代码依赖、医学知识）。

### Self-RAG

让 LLM 自我评估检索质量，决定是否需要再次检索：

```
检索 → 生成初步回答
  ↓
LLM 自评：回答是否充分？
  ├── 充分 → 输出
  └── 不充分 → 改写查询 → 再次检索 → 重新生成
```

### Multi-modal RAG

不只检索文本，还能检索图片、表格、代码：

```
文档包含 [文本] [图表] [代码]
  ↓
分别 Embedding：
  文本 → 文本向量
  图表 → 视觉模型 → 图像向量
  代码 → 代码模型 → 代码向量
  ↓
多模态检索 → 多模态 LLM（GPT-4V）生成回答
```

## 性能预算参考

| 环节 | 预期耗时 | 优化手段 |
|------|---------|---------|
| Query 改写 | 500-1000ms | 缓存常见查询 |
| Embedding | 50-100ms | 本地模型（我们用 Ollama ~50ms） |
| 向量检索 | 10-50ms | HNSW 索引（我们是 O(n) 遍历 ~5ms） |
| Reranker | 200-500ms | 控制候选数量 |
| LLM 生成（首字） | 500-2000ms | 流式输出 |
| **总计（首字延迟）** | **1.5-3.5s** | 并行 + 流式 |

## 技术栈总结

```
我们的实践项目技术栈：
  文档处理：自写 Markdown 加载 + cleanMarkdown() 清洗
  分块：    自写递归分块（800字 / 100重叠 / 80最小）
  Embedding：Ollama bge-m3（本地，1024 维，中文友好）
  向量存储：JSON 文件 + 余弦相似度遍历
  生成：    DeepSeek API + 流式输出 + Prompt 引用标注
  交互：    命令行 REPL

生产级升级路径：
  文档处理 → LangChain Loaders（多格式支持）
  分块     → LangChain RecursiveCharacterTextSplitter
  Embedding → OpenAI text-embedding-3-small 或保持 bge-m3
  向量存储 → PostgreSQL + pgvector + HNSW 索引
  检索     → 混合检索（向量 + BM25）+ Cohere Reranker
  评估     → RAGAS 自动评估管线
  可观测性 → Langfuse 追踪
```
