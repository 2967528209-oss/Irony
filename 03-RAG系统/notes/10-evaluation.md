# RAG 评估体系

## 为什么评估困难

传统软件：输入确定 → 输出确定 → 断言即可
RAG 系统：输入确定 → 输出不确定（LLM 生成） → 需要多维度评估

## 评估维度

RAG 的评估需要拆成两个独立环节：

```
用户问题 → 【检索环节】 → 检索结果 → 【生成环节】 → 最终回答

检索评估：检索到的文档是否相关？
生成评估：基于文档的回答是否准确？
```

### 检索质量指标

| 指标 | 含义 | 计算 |
|------|------|------|
| **Recall@K** | 正确文档出现在 Top-K 中的比例 | 命中数 / 总相关文档数 |
| **Precision@K** | Top-K 中相关文档的比例 | 相关数 / K |
| **MRR** | 第一个正确结果的排名倒数 | 1 / 排名 |
| **NDCG** | 归一化折损累积增益 | 考虑排名位置的综合指标 |

### 生成质量指标

| 指标 | 含义 | 评估方式 |
|------|------|---------|
| **Faithfulness** | 回答是否忠于检索文档（不编造） | LLM 判断回答是否能从文档推导 |
| **Answer Relevance** | 回答是否切中问题 | LLM 判断回答与问题的相关性 |
| **Context Relevance** | 检索上下文是否与问题相关 | LLM 判断文档与问题的相关性 |
| **Correctness** | 回答是否与标准答案一致 | 需要人工标注的 ground truth |

## RAGAS 框架

RAGAS 是最流行的 RAG 评估框架，用 LLM 自动评估：

```
评估数据集：
  question: "什么是年假天数？"
  ground_truth: "15天"              ← 人工标注的正确答案
  answer: "公司年假为15天"           ← RAG 系统的实际回答
  contexts: ["员工手册第4章..."]     ← 检索到的文档

RAGAS 自动计算：
  Faithfulness:      0.95  ← 回答忠于文档
  Answer Relevance:  0.90  ← 回答切中问题
  Context Precision:  0.85  ← 检索精度好
  Context Recall:     0.92  ← 检索召回高
```

## 评估管线架构

```
┌──────────────────────────────────────────────┐
│              评估数据集                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 问题集    │  │ 标准答案  │  │ 相关文档  │   │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘   │
│       │             │             │          │
├───────┼─────────────┼─────────────┼──────────┤
│       ↓             │             │          │
│  RAG 系统运行        │             │          │
│  ├── 检索结果 ───────┼──→ 检索指标评估        │
│  └── 生成回答 ───────┼──→ 生成指标评估        │
│                     ↓             ↓          │
│               ┌──────────────────────┐       │
│               │     评估报告          │       │
│               │  Faithfulness: 0.95  │       │
│               │  Recall@5:    0.90  │       │
│               │  Relevance:   0.88  │       │
│               └──────────────────────┘       │
└──────────────────────────────────────────────┘
```

## 评估驱动的优化循环

```
基线评估 → 发现检索 Recall 低
  → 调整分块大小 / 添加 Multi-query / 换 Embedding 模型
  → 重新评估 → Recall 提升
  → 发现 Faithfulness 低
  → 优化 Prompt / 减少检索噪音
  → 重新评估 → 全指标达标 ✓
```

## 架构决策

| 决策 | 建议 | 原因 |
|------|------|------|
| 评估框架 | RAGAS | 最成熟，LLM 自动评估 |
| 评估数据集 | 至少 50 条标注数据 | 太少不具统计意义 |
| 评估频率 | 每次管线变更后 | 防止优化一处破坏另一处 |
| 核心指标 | Faithfulness + Recall@5 | 分别代表生成和检索质量 |
