# 文本分块策略

## 为什么分块是 RAG 的关键

Embedding 模型有 token 限制（通常 512-8192），整篇文档无法直接编码。分块策略直接影响：

- **检索精度**：块太大 → 噪音多，相关性被稀释；块太小 → 缺乏上下文
- **生成质量**：上下文碎片化 → LLM 回答不完整
- **成本**：块越多 → Embedding 和存储成本越高

## 分块策略对比

### 1. 固定大小分块（Fixed Size）

```
文档 → 每 500 字符切一刀（重叠 50 字符）
```

| 优点 | 缺点 |
|------|------|
| 实现简单 | 可能切断句子/段落 |
| 块大小均匀 | 不尊重文档结构 |

适用：非结构化的长文本（日志、聊天记录）

### 2. 递归分块（Recursive）

```
优先按 "\n\n"（段落）分割
  → 超长段落按 "\n"（换行）分割
    → 仍然超长按 ". "（句子）分割
      → 最后按字符数分割
```

| 优点 | 缺点 |
|------|------|
| 尊重文档结构 | 块大小不均匀 |
| 保持语义完整 | 需要调参（分隔符优先级） |

**最常用的策略**，LangChain 的 `RecursiveCharacterTextSplitter` 即此方案。

### 3. 语义分块（Semantic）

```
逐句计算 Embedding → 相邻句子相似度低于阈值时切割
```

| 优点 | 缺点 |
|------|------|
| 语义最完整 | 计算成本高（需 Embedding 每个句子） |
| 自适应文档结构 | 速度慢 |

适用：对质量要求极高的场景

### 4. 按文档结构分块

```
Markdown：按标题层级（# / ## / ###）分割
代码：按函数/类分割（AST）
HTML：按 <section>/<article> 分割
```

| 优点 | 缺点 |
|------|------|
| 最尊重原始结构 | 仅适用于结构化文档 |
| 保留完整的知识单元 | 块大小差异大 |

## 关键参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `chunk_size` | 500-1000 字符 | 太小丢上下文，太大降精度 |
| `chunk_overlap` | 50-200 字符 | 块间重叠，防止关键信息被切断 |
| 分隔符优先级 | `\n\n` > `\n` > `. ` > 空格 | 递归分块的层级 |

## 决策流程

```
文档有明确结构（标题、代码函数）？
  → 是 → 按结构分块
  → 否 → 递归分块（默认首选）
           ↓
       质量不满意？
         → 尝试语义分块
```
