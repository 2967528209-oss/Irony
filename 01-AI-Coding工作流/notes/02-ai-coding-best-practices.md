# AI Coding 最佳实践

## 核心概念

### 人机协作模型

```
人类负责：需求定义、架构决策、质量把关、验收判断
AI 负责：代码实现、模式匹配、文档生成、重复劳动
```

核心原则：**人类做决策，AI 做执行**。

### 工作流闭环

```
需求分析 → 任务拆分 → Prompt 编写 → 代码生成 → 审查验证 → 迭代优化
    ↑                                                          |
    └──────────────── 反馈循环 ←────────────────────────────────┘
```

## 关键实践

### 1. 任务拆分

大任务拆成小任务，每个子任务：
- 有**单一明确的目标**
- 有**可验证的完成标准**
- 上下文在 AI 可处理范围内

### 2. Prompt 工程

结构化描述需求，包含四要素：
- **目标**：要做什么
- **上下文**：技术栈、已有代码、约束
- **具体要求**：功能 + 非功能需求
- **验收标准**：如何判断完成

### 3. 上下文管理

| 指令 | 用途 | 最佳时机 |
|------|------|----------|
| `@file` | 精确引用文件 | 修改/参考特定文件 |
| `@folder` | 了解模块结构 | 理解目录组织 |
| `@codebase` | 全项目搜索 | 查找用法/理解架构 |
| `@web` | 搜索网络 | 查最新 API/方案 |
| `@docs` | 引用文档 | 参考框架官方文档 |

**精准引用原则**：只给相关上下文，类型文件优先。

### 4. 验证闭环

每次生成代码后：
1. **阅读审查** — 理解代码逻辑
2. **类型检查** — `tsc --noEmit`
3. **运行测试** — 确保不破坏已有功能
4. **手动验证** — 实际运行看效果
5. **确认或迭代** — 不满意就进入下一轮

### 5. 多模型选择

- **复杂设计/Debug**：用最强模型（Opus）
- **日常编码**：用性价比模型（Sonnet）
- **快速补全**：用 Cursor Tab

### 6. 纪律

- 不盲目接受 AI 产出
- 对话过长时及时开新对话
- 一个功能验证通过再做下一个
- 重要决策记录在文档中

## 踩坑记录

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| AI 在长对话中偏离主题 | 上下文窗口污染 | 总结要点后开新对话 |
| 一次要求太多导致质量下降 | 单次 Prompt 信息过载 | 每轮迭代聚焦一个维度 |
| AI 生成的代码风格不一致 | 没有提供项目规范 | 用 Skills/Rules 约束 |
