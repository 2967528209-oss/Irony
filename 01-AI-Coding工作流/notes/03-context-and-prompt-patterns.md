# 上下文管理与 Prompt 模式

## 上下文管理策略

### 分层引用法

针对不同任务复杂度，采用不同的上下文引用策略：

**简单任务（改一个函数）**
```
@file 具体文件
```

**中等任务（加一个功能模块）**
```
@file 类型定义文件
@file 相关已有实现（作为参考）
```

**复杂任务（跨模块重构）**
```
@folder 涉及的模块目录
@file 核心类型定义
@file 入口文件
```

### 类型文件优先

TypeScript 项目中，优先引用 `*.types.ts` 文件。类型定义是最高密度的上下文信息，帮助 AI 理解数据结构和接口契约。

### 上下文清理

当对话过长时（> 20 轮），上下文窗口开始退化：
1. 总结当前进度和关键决策
2. 开启新对话
3. 用 `@file` 重新引入相关文件

## Prompt 模式

### 功能实现模式

```markdown
## 目标
实现 [功能名称]

## 上下文
- 技术栈：Next.js + TypeScript
- 已有：[列出相关已有代码]
- 约束：[列出技术约束]

## 要求
1. [功能要求]
2. [性能要求]
3. [安全要求]

## 验收标准
- [标准 1]
- [标准 2]
```

### Bug 修复模式

```markdown
## 问题
[具体错误现象]

## 预期行为
[应该怎样]

## 复现步骤
1. [步骤]
2. [步骤]

## 已尝试
- [尝试过的方案]
```

### 重构模式

```markdown
## 当前问题
[为什么要重构]

## 改进方向
[想要达到的效果]

## 约束
- 不改变公共 API
- 保持向后兼容
- 测试不能挂
```

### 代码审查模式

```markdown
请审查以下代码，关注：
1. 逻辑正确性
2. 错误处理完整性
3. 安全风险
4. 性能问题
5. 是否符合项目规范（@file .cursor/skills/coding-standards/SKILL.md）
```

## 迭代策略

分轮迭代，每轮聚焦一个维度：

| 轮次 | 聚焦点 | Prompt 关键词 |
|------|--------|--------------|
| 第 1 轮 | 核心功能 | "先实现基础功能" |
| 第 2 轮 | 边界处理 | "补充错误处理和边界条件" |
| 第 3 轮 | 质量优化 | "优化代码质量和可读性" |
| 第 4 轮 | 测试 | "为以上实现编写测试" |

避免一次性在 Prompt 中要求所有维度，这会导致 AI 注意力分散、产出质量下降。
