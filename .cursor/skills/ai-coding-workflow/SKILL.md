# AI 编码工作流

## 核心工作流

```
需求分析 → 任务拆分 → Prompt 编写 → 代码生成 → 审查验证 → 迭代优化
```

每一步都遵循「人类决策 + AI 执行」原则。

## 任务拆分方法论

### 拆分原则

1. **单一职责**：每个子任务只完成一个明确的功能点
2. **可验证性**：每个子任务完成后有明确的验证标准
3. **渐进式**：从核心功能开始，逐步扩展
4. **上下文可控**：单个子任务的上下文不超过 AI 有效处理范围

### 拆分粒度

| 规模 | 拆分策略 | 示例 |
|------|----------|------|
| 小功能 | 1-2 个 Prompt 完成 | 添加一个工具函数 |
| 中等功能 | 3-5 个子任务 | 实现一个 API 端点（路由+服务+模型+测试） |
| 大功能 | 拆为多个中等功能 | 实现用户认证系统 |
| 系统级 | 先做架构设计，再逐模块实现 | 构建 RAG 系统 |

## Prompt 工程模式

### 结构化需求描述

```markdown
## 目标
[一句话说明要实现什么]

## 上下文
[相关的技术栈、已有代码、约束条件]

## 具体要求
1. [功能要求 1]
2. [功能要求 2]
3. [非功能要求：性能/安全/兼容性]

## 验收标准
- [如何判断完成]
```

### 常用 Prompt 模式

| 模式 | 适用场景 | 关键要素 |
|------|----------|----------|
| 功能实现 | 新增功能 | 目标 + 输入/输出 + 约束 |
| Bug 修复 | 修复问题 | 现象 + 预期行为 + 复现步骤 |
| 重构优化 | 改善代码 | 当前问题 + 改进方向 + 不变约束 |
| 代码审查 | 质量检查 | 关注维度 + 严格程度 |
| 架构设计 | 系统设计 | 需求 + 约束 + 技术偏好 |

### Prompt 迭代策略

- **第一轮**：描述核心需求，获取基础实现
- **第二轮**：基于产出补充边界条件和错误处理
- **第三轮**：优化代码质量、性能、可读性
- 每轮迭代聚焦一个维度，避免一次性要求太多

## 上下文管理

### 引用策略

| 指令 | 用途 | 使用时机 |
|------|------|----------|
| `@file` | 引用具体文件 | 需要 AI 理解特定文件内容 |
| `@folder` | 引用目录 | 需要 AI 了解模块结构 |
| `@web` | 搜索网络 | 查询最新 API/文档/解决方案 |
| `@docs` | 引用文档 | 参考框架官方文档 |
| `@codebase` | 全项目搜索 | 理解全局架构或查找用法 |

### 上下文最佳实践

1. **精准引用**：只引用相关文件，不给无关上下文
2. **分层引用**：先给架构概览（`@folder`），再聚焦具体文件（`@file`）
3. **类型文件优先**：引用类型定义帮助 AI 理解数据结构
4. **示例驱动**：提供已有的类似实现作为参考

## 验证闭环

每次代码生成后执行：

```
生成代码 → 阅读审查 → 类型检查 → 运行测试 → 手动验证 → 确认/迭代
```

### 审查要点

1. **逻辑正确性**：是否实现了预期功能
2. **类型安全**：类型定义是否完整准确
3. **错误处理**：是否覆盖异常场景
4. **代码风格**：是否符合项目编码规范
5. **安全性**：是否有注入、泄露等风险

## 多模型切换策略

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 复杂架构设计 | Claude Opus / GPT-4o | 深度推理能力强 |
| 日常编码实现 | Claude Sonnet / GPT-4o-mini | 性价比高，速度快 |
| 快速补全/小改动 | Cursor Tab / Copilot | 即时反馈 |
| 文档撰写 | Claude Sonnet | 中文表达自然 |
| Debug 分析 | Claude Opus | 长上下文理解强 |

## AI Coding 纪律

1. **不盲目接受**：每次生成的代码都必须审查理解
2. **保持上下文纪律**：发现 AI 偏离时及时纠正或重启对话
3. **先验证再继续**：一个功能验证通过后再做下一个
4. **记录决策**：重要的架构决策和取舍在文档中记录
5. **定期整理**：对话过长时总结要点开启新对话
