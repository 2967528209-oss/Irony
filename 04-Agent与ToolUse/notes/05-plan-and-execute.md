# Plan-and-Execute 模式

## 与 ReAct 的对比

ReAct 是**边想边做**——每一步推理决定下一步行动，没有全局规划。

Plan-and-Execute 是**先规划后执行**——先生成完整计划，再逐步执行。

```
ReAct：
  Step 1: 想 → 做
  Step 2: 想 → 做   ← 每步独立决策，可能偏离
  Step 3: 想 → 做

Plan-and-Execute：
  Plan: [任务1, 任务2, 任务3]  ← 先出完整方案
  Execute: 任务1 → 任务2 → 任务3  ← 按方案执行
  Replan: 如果中间发现计划有问题 → 调整计划
```

## 适用场景

| 模式 | 适用 | 不适用 |
|------|------|--------|
| **ReAct** | 探索性任务、信息不确定 | 多步依赖的复杂任务 |
| **Plan-and-Execute** | 步骤明确的多步任务 | 简单直接的问题 |

例子：

```
"查一下今天天气" → ReAct（一步搞定）

"帮我做一个市场调研报告：
 1. 搜索行业数据
 2. 分析竞争对手
 3. 总结趋势
 4. 生成报告" → Plan-and-Execute（多步、有序）
```

## 架构

```
用户任务
  ↓
┌─────────────┐
│  Planner     │  ← LLM 调用 1：生成计划
│  输出计划列表 │
└──────┬──────┘
       ↓
┌─────────────┐
│  Executor    │  ← LLM 调用 2~N：逐步执行
│  执行当前步骤 │
│  更新状态     │
└──────┬──────┘
       ↓
  所有步骤完成？
  ├── 否 → Replan（可选：根据执行情况调整计划）
  └── 是 → 汇总结果，输出最终回答
```

## 两个 LLM 角色

Plan-and-Execute 通常用**两个不同的 Prompt**（可以是同一个模型）：

### Planner（规划者）
```
你是一个任务规划师。给定用户任务，将其分解为有序的子任务列表。
每个子任务应该是一个可以独立执行的步骤。
输出 JSON 数组：[{"step": 1, "task": "描述"}, ...]
```

### Executor（执行者）
```
你是一个任务执行者。执行以下任务，使用可用的工具完成。
当前任务：{current_step}
之前步骤的结果：{previous_results}
```

## 实际例子

```
用户: "调研一下 RAG 系统的技术栈选择，然后计算一个中等规模系统的月成本"

Planner 输出:
  Step 1: 搜索 RAG 系统推荐的技术栈
  Step 2: 搜索各技术组件的定价信息
  Step 3: 计算中等规模（10 万文档、每天 1000 次查询）的月成本
  Step 4: 汇总整理成报告

Executor 逐步执行:
  Step 1: search_knowledge("RAG 技术栈") → 得到推荐方案
  Step 2: search_knowledge("向量数据库定价") → 得到价格信息
  Step 3: calculate("...") → 得到月成本
  Step 4: 基于前面结果，生成汇总报告
```

## 在 Cursor 中的体现

Cursor Agent 的实际行为更接近 **ReAct + 隐式 Plan** 的混合：
- 复杂任务时，它会先输出"我的计划是..."（类似 Plan）
- 然后逐步执行，但每步仍然有 Think/Act 的 ReAct 决策
- 如果中间步骤失败，会自适应调整

## 与 ReAct 的权衡

| 维度 | ReAct | Plan-and-Execute |
|------|-------|-----------------|
| 灵活性 | 高（每步可调整） | 中（计划约束执行） |
| 可预测性 | 低（不知道走几步） | 高（计划可提前审查） |
| 多步一致性 | 弱（容易偏离） | 强（计划保持方向） |
| 成本 | 不确定 | 较可控（步数已知） |
| 实现复杂度 | 低 | 中（需要 Planner + Executor） |
